## ДЗ2. MVP проекта

**Проект:** NLP-генератор медицинских SOAP-протоколов

---

## 1. Цель этапа

Целью второго этапа была реализация MVP-версии проекта, которая воспроизводит основной функционал системы, то есть преобразование текстового диалога врача и пациента в структурированный медицинский протокол по стандарту SOAP.

На данном этапе важно было не построить production-ready систему, а подтвердить, что ключевая идея проекта жизнеспособна технически, что пользователь может передать текстовый диалог, а система способна вернуть результат в формате S / O / A / P.

---

## 2. Что было реализовано

В рамках этапа была собрана рабочая MVP-версия проекта со следующими компонентами:

### 2.1. Пользовательский интерфейс

Реализован веб-интерфейс на Streamlit, который позволяет:

- вставить текст диалога врача и пациента  
- выбрать тестовый сценарий из заранее подготовленного набора  
- выбрать режим генерации:
  - LLM (основной режим)
  - Baseline fallback
- получить результат в формате SOAP  
- скачать результат в виде `.txt` файла  

### 2.2. Baseline-режим

Реализован rule-based baseline, который:

- разделяет реплики врача и пациента  
- определяет секции S, O, A, P по ключевым словам и простым эвристикам  
- используется как fallback-режим и контрольная версия  

### 2.3. LLM-режим

Реализован основной LLM-режим генерации SOAP через локальную модель, запущенную через Ollama.  
В качестве модели использовалась Qwen.

LLM-режим:

- принимает исходный текст диалога  
- получает инструкцию и few-shot пример  
- возвращает структурированный SOAP в JSON-формате  
- затем результат проходит постобработку и отображается в интерфейсе  

### 2.4. Backend API

Дополнительно реализован backend на FastAPI, который содержит:

- GET /health — проверка доступности backend  
- POST /generate — генерация SOAP через baseline  
- POST /generate_llm — генерация SOAP через LLM  

Это позволило разделить пользовательский интерфейс и сервисный слой.

### 2.5. Тестовые данные и эталоны

Подготовлен набор тестовых сценариев по нескольким типам клинических ситуаций:

- ОРВИ  
- боль в животе  
- головная боль и давление  
- аллергическая реакция  
- боль в спине  
- короткий диалог  
- кейс без объективных данных  
- кейс без рекомендаций  

Для этих сценариев были подготовлены gold-эталоны SOAP, что позволило добавить базовую автоматическую оценку качества.

### 2.6. Оценка качества

В MVP был реализован блок оценки качества, включающий:

- ROUGE-L — сравнение предсказанного SOAP с эталонным  
- Entity Coverage — оценка покрытия ключевых сущностей по секциям  

Блок оценки качества спрятан в интерфейсе в отдельный раскрывающийся раздел, чтобы не перегружать основной пользовательский экран.

## 3. Почему были выбраны именно такие решения

### 3.1. Почему Streamlit

Streamlit был выбран как быстрый инструмент для сборки MVP-интерфейса, поскольку:

- он позволяет быстро получить рабочее веб-приложение  
- не требует сложной frontend-разработки  
- подходит для демонстрации прототипа  
- хорошо подходит для учебного проекта, где важнее core functionality, чем production UI  

### 3.2. Почему FastAPI

FastAPI был добавлен как минимальный backend-слой, потому что:

- он хорошо соответствует архитектурной логике проекта  
- позволяет вынести генерацию SOAP в отдельный API  
- это создаёт задел под следующий этап, связанный с инфраструктурой и деплоем  
- наличие backend-а делает проект ближе к реальной сервисной архитектуре  

### 3.3. Почему baseline + LLM

На этапе MVP было решено реализовать два режима генерации.

#### Baseline fallback

Rule-based baseline был добавлен потому, что:

- он прост и интерпретируем  
- позволяет быстро проверить работоспособность core logic  
- служит контрольной версией  
- используется как резервный вариант, если LLM-режим недоступен  

#### LLM (основной режим)

В рамках данной задачи LLM используется не для генерации новых медицинских фактов, а для:

- извлечения информации из диалога  
- структурирования текста по формату SOAP  
- более гибкой обработки свободных формулировок  
- нормализации текста в сторону более медицинского стиля  

То есть, baseline в проекте играет роль fallback и контрольной версии, а LLM — роль основного NLP-ядра.

### 3.4. Почему локальная модель, а не внешний API

Изначально рассматривался вариант использования облачного API, однако в процессе работы возникли ограничения доступа по региону.  
В результате было принято решение перейти на локальный запуск модели через Ollama.

Это решение оказалось удачным по нескольким причинам:

- не требуется внешний API-ключ  
- отсутствует зависимость от облачного сервиса  
- медицинские данные не отправляются во внешний API  
- такой вариант лучше соответствует требованиям конфиденциальности  
- архитектура становится ближе к идее локальной обработки данных  

---

## 4. Какие трудности возникли в процессе

### 4.1. Ошибки baseline-логики

На этапе разработки baseline-режима были замечены типичные ошибки:

- вопросы врача попадали в секцию O  
- строки с планом лечения могли попадать в секцию A  
- часть текста могла некорректно объединяться в одну секцию  

Проблема решалась поэтапным улучшением правил:

- добавлена фильтрация вопросов  
- разнесены логики определения A, P и O  
- улучшены словари ключевых признаков  

### 4.2. Перегрузка пользовательского интерфейса

Изначально эталонный SOAP и блок оценки качества отображались прямо на основном экране, что делало интерфейс перегруженным.  
Позже было принято решение спрятать этот блок в expander, чтобы основной экран показывал только результат, а служебная оценка качества была доступна по запросу.

---

## 5. Что получилось в результате

В результате была собрана рабочая MVP-система, которая:

- принимает текстовый диалог  
- преобразует его в SOAP  
- поддерживает baseline и LLM-режимы  
- содержит тестовые клинические кейсы  
- имеет backend API  
- позволяет автоматически оценивать качество на эталонных примерах  

По сути, на данном этапе удалось реализовать core functionality проекта и подтвердить, что идея автоматического формирования SOAP-протокола из диалога действительно может быть реализована в рабочем виде.

---

## 6. Что пока остаётся ограничением MVP

Несмотря на работоспособность прототипа, у текущей версии есть ограничения:

- система работает только с текстом  
- аудио и speech-to-text не реализованы  
- медицинская терминологическая нормализация ограничена  
- метрики качества являются упрощёнными и не заменяют экспертную медицинскую оценку  

Также стоит отметить, что backend и frontend пока работают как отдельные части MVP, без полноценной production-связки и инфраструктурной обвязки. Это планируется как следующий шаг развития проекта.

---

## 7. Почему результат этапа можно считать успешным

Этап можно считать успешным, потому что выполнены основные требования MVP:

- реализован рабочий код проекта  
- реализован пользовательский интерфейс  
- реализован сервисный backend  
- реализован LLM-режим  
- реализован fallback  
- подготовлены тестовые кейсы и эталоны  
- есть базовая оценка качества  
- проект оформлен в структуре, пригодной для дальнейшего развития  

С точки зрения бизнес-идеи MVP подтверждает, что система действительно может сократить объём ручной работы по структурированию текста медицинского приёма.
