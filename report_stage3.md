# Отчёт по выполнению ДЗ №3: Инфраструктура проекта

## 1. Описание проделанной работы
На данном этапе целью было превратить MVP-скрипт в устойчивую сервисную инфраструктуру. Основной акцент сделан на контейнеризации, автоматизации развертывания и внедрении слоя хранения данных.

## 2. Технические решения и стек
- **Docker Compose:** Использован для объединения двух сервисов: `app` (основное приложение) и `db` (база данных PostgreSQL). Это решило проблему «конфликта окружений».
- **PostgreSQL 15:** Выбрана как надежное реляционное хранилище. Внедрение БД позволило реализовать логирование запросов (диалог, результат, режим генерации, метка времени).
- **SQLAlchemy:** Использована для управления схемами данных. Таблицы инициализируются автоматически при первом запуске контейнера.
- **Интеграция с LLM:** Реализован механизм обращения к локальному серверу Ollama через `host.docker.internal`. Это критически важное решение для медицины, так как данные обрабатываются внутри локального контура без отправки в облако.
- **Мониторинг:** Реализована проверка доступности сервиса через эндпоинт /health (FastAPI) и визуализация статуса подключения к БД в интерфейсе Streamlit.

## 3. Сложности и их решение
В процессе работы мы столкнулись с несколькими критическими проблемами:
1. **Сетевой мост (Network Bridge):** Контейнер не видел Ollama на порту 11434 хоста. 
   - *Решение:* Добавление `extra_hosts` в docker-compose и настройка переменной окружения `OLLAMA_HOST` на стороне сервера.
2. **Ошибка 404 (Model Not Found):** Возникала из-за несовпадения имен моделей в коде и в локальном хранилище Ollama.
   - *Решение:* Синхронизация названий (`qwen2.5:3b`) через переменные окружения в `docker-compose.yaml`.
3. **Зависимость сервисов:** Приложение падало, если база данных не успевала запуститься.
   - *Решение:* Использование механизма `depends_on` и доработка логики подключения в коде.

## 4. Оценка использования LLM и стоимость
Для разработки инфраструктуры использовались кодовые ассистенты (LLM).
- **Время:** Использование моделей позволило сократить время настройки Docker-сетей и SQL-схем с 6-8 часов до 4 часов. Модели помогали в отладке ошибок сетевого взаимодействия контейнеров.
- **Стоимость:** 0$. Все инструменты (Docker, PostgreSQL, Ollama, Qwen) являются open-source. Использовались бесплатные лимиты генеративных моделей для помощи в написании кода.

## 5. Заключение
Инфраструктурная обвязка сделала проект готовым к эксплуатации. Теперь сервис поддерживает историю запросов, легко разворачивается на любом сервере с Docker и сохраняет полную конфиденциальность данных пациентов за счет локального контура обработки.
