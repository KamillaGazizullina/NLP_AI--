# NLP-генератор медицинских SOAP-протоколов

## О проекте

Этот проект преобразует текстовый диалог врача и пациента в структурированный медицинский протокол по стандарту **SOAP**:

- **S — Subjective**: жалобы и ощущения пациента  
- **O — Objective**: объективные данные осмотра  
- **A — Assessment**: предварительное медицинское заключение  
- **P — Plan**: рекомендации и план действий  

MVP реализован как веб-приложение на **Streamlit** с двумя режимами генерации:

- **LLM (основной режим)** - локальная генерация SOAP с помощью **Ollama + Qwen**
- **Baseline fallback** - rule-based извлечение по эвристикам и ключевым признакам

Также в проекте реализован отдельный backend на **FastAPI** и блок базовой оценки качества по эталонным кейсам.

---

## Бизнес-задача

Врачи-терапевты тратят значительное время на заполнение медицинской документации во время или после приёма. Даже если диалог врача и пациента уже доступен в текстовом виде, врачу всё равно приходится вручную оформлять итоговый отчёт.

Цель проекта: ускорить создание медицинской записи и уменьшить нагрузку на врача за счёт автоматического преобразования диалога в SOAP-протокол.

---

## Что умеет MVP

- принимает текстовый диалог врача и пациента
- преобразует его в SOAP-структуру
- поддерживает **два режима генерации**:
  - локальный **LLM-режим**
  - **baseline fallback**
- позволяет выбрать готовые тестовые сценарии
- показывает результат в интерфейсе Streamlit
- позволяет скачать итоговый SOAP как `.txt`
- содержит **gold-эталоны** для тестовых кейсов
- считает простые метрики качества:
  - **ROUGE-L**
  - **Entity Coverage**
- предоставляет backend API на FastAPI:
  - `/health`
  - `/generate`
  - `/generate_llm`

---

## Архитектура MVP

### Пользовательский сценарий
`Диалог врача и пациента -> Streamlit UI -> Core генерации -> SOAP`

### Основные компоненты

- **Streamlit** - пользовательский интерфейс
- **FastAPI** - backend API
- **Baseline pipeline** - rule-based разметка по секциям SOAP
- **LLM pipeline** - локальная генерация через Ollama/Qwen
- **Metrics** - оценка качества по эталонным кейсам

### Логика обработки
1. Пользователь вставляет текст диалога или выбирает тестовый кейс
2. Выбирает режим генерации:
   - LLM
   - Baseline fallback
3. Система формирует SOAP
4. При наличии gold-эталона можно раскрыть блок оценки качества
5. Результат можно скачать как `.txt`

## Как запустить проект

### 1. Установить зависимости Python

    py -m pip install -r requirements.txt

### 2. Установить Ollama

Скачать и установить Ollama для Windows с официального сайта.

После установки проверить:

    ollama --version

### 3. Скачать локальную модель

Например, используем:

    ollama pull qwen3:4b

При необходимости можно использовать более крупную модель:

    ollama pull qwen3:8b

### 4. Настроить `.env`

Создать файл `.env` в корне проекта:

    OLLAMA_MODEL=qwen3:4b
    OLLAMA_BASE_URL=http://localhost:11434/api/chat

### 5. Запустить Streamlit UI

    py -m streamlit run app/ui.py

После запуска откроется веб-интерфейс в браузере.

### 6. Запустить FastAPI backend

В отдельном терминале:

    py -m uvicorn app.api:app --reload

Проверить backend можно по адресам:

- Health check: http://127.0.0.1:8000/health  
- Swagger UI: http://127.0.0.1:8000/docs

## API endpoints

### GET /health

Проверка доступности backend-а.

Ответ:

    {"status": "ok"}

### POST /generate

Генерация SOAP через baseline fallback.

Пример запроса:

    {
      "dialog": "Пациент: Уже 3 дня температура 38, кашель..."
    }

### POST /generate_llm

Генерация SOAP через локальный LLM.

Пример запроса:

    {
      "dialog": "Пациент: Уже 3 дня температура 38, кашель..."
    }

## Оценка качества

В MVP реализован скрытый блок оценки качества по тестовым кейсам.

Используются:

- **ROUGE-L**  
  Оценивает совпадение предсказанного SOAP с эталонным SOAP на уровне токенов  

- **Entity Coverage**  
  Оценивает покрытие ключевых сущностей по секциям SOAP  

Метрики отображаются в интерфейсе внутри раскрывающегося блока:  
**«Показать эталонный SOAP и оценку качества»**

---

## Почему в проекте есть два режима

### LLM (основной режим)

Используется как основное NLP-ядро проекта.  
Позволяет:

- лучше работать со свободными формулировками  
- аккуратнее структурировать текст  
- выдавать более медицински оформленный результат  

### Baseline fallback

Нужен как:

- интерпретируемый резервный режим  
- контрольная версия  
- способ сравнения с LLM  

---

## Ограничения MVP

- проект работает только с текстом  
- аудио и speech-to-text не входят в рамки MVP  
- в MVP не используется база знаний и retrieval  
- медицинская терминологическая нормализация реализована ограниченно  
- оценка качества является упрощённой и не заменяет медицинскую экспертизу  
- проект не предназначен для постановки окончательного диагноза без врача  

---

## Безопасность и конфиденциальность

В MVP выбран локальный LLM через Ollama, это позволяет не отправлять медицинский текст во внешний облачный API. Это лучше соответствует требованиям конфиденциальности и теме обработки чувствительных медицинских данных.

## Технологический стек
- **Frontend:** Streamlit (удобный интерфейс для врача)
- **Backend:** FastAPI (программный доступ через API)
- **LLM Core:** Локальный запуск Qwen 2.5 через Ollama (конфиденциальность данных)
- **Infrastructure:** Docker & Docker Compose (оркестрация сервисов)
- **Storage:** PostgreSQL (хранение истории генераций и логов)
- **ORM:** SQLAlchemy (управление базой данных из Python)

## Возможности инфраструктуры (Этап 3)
В рамках третьего этапа проект был переведен на сервисную архитектуру:
1. **Контейнеризация:** Приложение полностью изолировано в Docker. Запуск всей системы (UI + API + DB) выполняется одной командой.
2. **База данных:** Интегрирована PostgreSQL для логирования транзакций. Теперь каждый запрос врача сохраняется, что позволяет проводить аудит качества.
3. **Сетевое взаимодействие:** Настроена связь между контейнерами и хостовой машиной для работы с локальной LLM.
